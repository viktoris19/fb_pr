1) Практика 9 — Refresh-токены

Цель
	1.	При входе (POST /api/auth/login) возвращать пару токенов:

{ "accessToken": "...", "refreshToken": "..." }

	2.	Реализовать маршрут:

	•	POST /api/auth/refresh — обновляет пару токенов и возвращает новую пару:

{ "accessToken": "...", "refreshToken": "..." }

Как это работает
	•	accessToken живёт недолго → им подписываем запросы к защищённым маршрутам.
	•	refreshToken живёт дольше → нужен, чтобы автоматически получать новый accessToken без повторного логина.

Где смотреть в проекте
	backend/routes/auth.js
	•	login возвращает пару токенов
	•	refresh выдаёт новую пару токенов
	backend/middleware/authJwt.js
	•	проверка access-токена (Bearer)
	•	константы/секреты/время жизни токенов (учебные)

Немного теории для закрепления


1) Где у вас создаются токены

Токены создаются только там, где вызывается jwt.sign(...)

В вашем проекте jwt.sign(...) вызывается в двух функциях:
	•	generateAccessToken(user)
	•	generateRefreshToken(user)

Обе функции лежат в backend/routes/auth.js.

А где они вызываются (то есть где реально “выпускаются” токены)?
	•	в POST /api/auth/login
→ выдаём первую пару { accessToken, refreshToken }
	•	в POST /api/auth/refresh
→ выдаём новую пару { accessToken, refreshToken } (после проверки refresh)

То есть:

создание = jwt.sign → функции generate… → вызываются в login и refresh.

⸻

2) Почему это “раскидано” между middleware и роутами

Это нормальная архитектура Express. Разделение по ответственности:

Роуты (routes) = “бизнес-операции”

auth.js отвечает за:
	•	выдать токены (login)
	•	обновить токены (refresh)
	•	зарегистрировать (register)

Почему именно тут:
	•	токен выдаётся в ответ на конкретную операцию (login/refresh)
	•	это часть бизнес-логики аутентификации

Middleware = “фильтр/охранник”

authJwt.js отвечает за:
	•	проверить accessToken на входе в защищённые роуты

Почему именно тут:
	•	проверка должна быть одинаковая для любых защищённых эндпоинтов (/me, товары и т.д.)
	•	если писать проверку в каждом роуте — получите копипаст и ошибки

⸻

3) Модель “конвейера”

Представьте 2 ситуации:

Ситуация A: “получить токены”

Клиент делает:
	•	POST /api/auth/login

Сервер:
	•	попадает в роут login
	•	создаёт токены (jwt.sign) → отдаёт клиенту

Здесь middleware не нужен, потому что логин — это публичная операция.

⸻

Ситуация B: “доступ к защищённому ресурсу”

Клиент делает:
	•	GET /api/auth/me
	•	или защищённый /api/products/:id

Сервер:
	1.	сначала запускает authMiddleware
	2.	middleware читает Authorization: Bearer ...
	3.	делает jwt.verify(accessToken, JWT_SECRET)
	4.	если ок → кладёт req.user и пропускает дальше (next())
	5.	дальше выполняется обработчик роута (например /me) и использует req.user.sub

То есть:
	•	токен создаётся в роуте
	•	токен проверяется в middleware
	•	роут получает результат проверки через req.user

Это и есть “раскидано”, но это правильное разделение.

⸻

4) Почему refreshToken не проверяется middleware?

Потому что refreshToken:
	•	не даёт доступ к ресурсам
	•	нужен только для /refresh

Поэтому логика refresh проверки живёт внутри POST /refresh, а не в общем middleware.
