## Практика 7. Базовые методы аутентификации (bcrypt + соль)

### Цель

1. Реализовать **регистрацию** пользователя с хранением пароля в виде **bcrypt-хеша**.
2. Реализовать **вход** (проверка пароля через `bcrypt.compare`).
3. Подготовить API и протестировать в Postman (или через `backend/api.http`).

### Маршруты (минимум)

- `POST /api/auth/register` — регистрация пользователя
- `POST /api/auth/login` — вход

### Поля пользователя (минимум)

- `id`
- `email` (логин)
- `first_name`
- `last_name`
- `password` (на сервере хранить **только хеш**)

## Практика 8. JWT-токены (выдача, проверка, защищённые маршруты)

### Цель

1. При входе выдавать **JWT accessToken**.
2. Реализовать middleware `authMiddleware`, который проверяет `Authorization: Bearer <token>`.
3. Добавить защищённый маршрут:

- `GET /api/auth/me` — вернуть объект текущего пользователя (по токену)

### Защитить маршруты товаров

- `GET /api/products/:id`
- `PUT /api/products/:id`
- `DELETE /api/products/:id`

## Какие зависимости добавились в проекте

Основан на репозитории проекта для Лекции 2 https://github.com/ZagorodnikhNick/fb-express-products-swagger

bcrypt

Зачем: безопасное хеширование паролей.
Где: routes/auth.js:
• bcrypt.hash(password, 10) — сохранить пароль как хеш
• bcrypt.compare(password, hash) — проверить пароль при логине
Что даёт: пароль в базе/памяти хранится не открытым текстом, а как хеш (с солью внутри).

jsonwebtoken

Зачем: выпуск и проверка JWT токенов.
Где:
• routes/auth.js: jwt.sign(...) — выдача accessToken при логине
• middleware/authJwt.js: jwt.verify(...) — проверка токена на защищённых маршрутах
Что даёт: механизм “вошёл → получил токен → ходишь на защищённые эндпоинты”.

## Как протестировать

### Вариант A — Postman

1. `POST /api/auth/register`
2. `POST /api/auth/login` → заберите `accessToken`
3. В Postman добавьте заголовок:
   - `Authorization: Bearer <accessToken>`
4. Проверьте:
   - `GET /api/auth/me`
   - защищённые `GET/PUT/DELETE /api/products/:id`

### Вариант B — VS Code REST Client

Откройте `backend/api.http` и выполняйте запросы.

## Фронтенд (рекомендуется) Реализовать страницу регистрации и входа на фронтенде

Цель: научиться работать с API аутентификации из React.

Что сделать на фронте (минимум):

1. Страница/форма Регистрация:
   • поля: email, first_name, last_name, password
   • запрос: POST /api/auth/register
   • обработка ошибок (показать сообщение пользователю)

2. Страница/форма Вход:
   • поля: email, password
   • запрос: POST /api/auth/login
   • получить accessToken и сохранить:
   • минимум: в useState
   • рекомендуется: в localStorage (чтобы не слетал при обновлении страницы)

3. После входа сделать кнопку “Проверить профиль”:
   • запрос: GET /api/auth/me
   • передать заголовок:
   • Authorization: Bearer <accessToken>

4. Для защищённых товаров (GET/PUT/DELETE /api/products/:id) тоже передавать токен в заголовке.

Техническая подсказка:
• token хранить в одном месте (например AuthContext или общий apiClient), чтобы не копипастить заголовки.
